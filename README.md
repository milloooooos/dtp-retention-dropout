# DTP 患者服务 · 完整复盘分析体系

上传 **销售底表**、**随访任务表**（可选）、**项目药房 TOP 清单**（可选），自动输出 DTP 药房患者管理的完整复盘体系：

- 留存率 Cohort（两套口径并存）
- 脱落率 A（滚动）/ 脱落率 B（累计沉默）
- DOT 分解（老患 / 新患窗口 / 重叠患者，同比）
- 新患月度趋势 & 项目药房新患变化
- 复购率分解（全量 vs 老患 + 新患稀释占比）
- 医院维度 / 医生维度 / 项目药房维度
- 异常医院拖后腿医生钻取
- 半年最该沟通清单 + 改进措施与责任
- 跨表关联：销售脱落患者 ↔ 随访脱落原因（修正后的细分类框架 + 双视角 + 覆盖率 + 低匹配报警）

最终产出 **Excel 完整报告** + **Word 复盘报告**，可直接用于管理层汇报。

> 面向 AZ（阿斯利康）等药企委托 DTP 药房的患者服务复盘场景。数据全程在本地/浏览器处理，不上传第三方。

## 功能模块

| 模块 | 口径 | 说明 |
|------|------|------|
| **留存率 Cohort（两套口径）** | 按首购月分群（=新患队列） | ①仅看购药时间：Mk=第k月当月有购药占比；②结合说明书盒数覆盖：购药覆盖=盒数×每盒天数，覆盖到第k月即算留存。两套并存，差值≈囤药/断续强度 |
| **脱落率 A（滚动）** | 基准月 M-2 有购药、M-1∪M 无购药 → 脱落 | 月度口径，适合看趋势 |
| **脱落率 B（累计沉默）** | 末次购药距数据终点 > N×用药间隔 → 真停药 | 另给「已观察%」剔除新近患者右删失；倍数=3 时≈药吃完后再空2个周期 |
| **DOT 分解** | 窗口内购买总盒数 / 去重患者数（盒/人） | 拆为全量/老患/新患窗口/重叠患者，本期 vs 同比，区分真实持续性改善与构成污染 |
| **新患结构** | 首购月 = 当月新患 | 月度趋势、项目药房新患变化、新患稀释占比 |
| **医院/医生/药房维度** | 项目药房范围内 | 识别高脱落、低 DOT、新患致低 DOT、大店低 DOT 等重点对象 |
| **钻取** | 异常医院 → TOP5 拖后腿医生 | 下钻到具体医生，便于院内外联动 |
| **沟通清单 & 改进措施** | 自动汇总重点关注对象 | 医院/医生/药房维度 + 通用改进措施与责任方 |
| **跨表关联** | 销售脱落患者 ↔ 随访脱落原因 | 品牌层面关联（可靠）+ 患者级（姓名匹配 + 药房软对齐置信度）；脱落原因用修正后的细分类框架 |
| **脱落原因分类（修正）** | 随访「未按计划持续用药原因」等 → 固定框架 | 复用 `churn_logic`：10 细类 + 一级(医生/患者/其他)，四原则(精确优先→关键词兜底→自上而下分区互斥→统一兜底)，医嘱原因不漏给"患者自主停药" |
| **患者级双视角** | 同一患者多条随访记录 | 并列「最新一次」(当前状态) 与「最近一次有原因」(有记载的脱落原因)，差值处即"当前正常但曾有脱落原因" |
| **覆盖率 / 空原因** | 分母(销售脱落B) × 分子(随访匹配) | 空原因≠无脱落，单独计数不混入任何细类；匹配率<30% 红色报警"可能传错底表" |
| **自丰富品牌映射表** | 药名 → 品牌 | 品牌字直匹配 → 通用名别名匹配 → 从数据自动学习新别名写回 `brand_aliases.json` |

## 快速开始（本地）

```bash
pip install -r requirements.txt
streamlit run app.py
# 浏览器打开 http://localhost:8501
```

1. 上传「销售底表」（xlsx/xls）
2. 可选上传「随访任务表」（可多选，如 2025H1 + 2026H1 两个 xls）
3. 可选上传「项目药房 TOP 清单」（含 药房名称 + 各品种 TOP/项目/重点列 + 城市）
4. 调侧边栏参数（留存追踪月数、脱落率B倍数、是否输出近似患者级）
5. 点「运行分析」→ 看 10 个标签页 → 下载 Excel / Word 报告

> 不传随访表也能算留存率、脱落率、DOT、新患、维度分析；不传 TOP 清单则默认全部药房为项目。

## 数据要求

**销售底表**（Sheet 含以下列，列名可用等价别名，引擎自动识别）：
`商品名称`（含品牌括号，如"甲磺酸奥希替尼片(泰瑞沙)"）、`销售时间`、`销售数量`、`患者ID`（或会员号/开票抬头）、`会员姓名`（可选）、`药房名称`、`医疗单位`、`处方医生`。

**随访任务表**（xls，历史任务导出）：
`药品名称`、`患者`（姓名）、`用药周期状态`、`未按计划持续用药原因`、`本月未购药的原因`、`患者首购日期`、`末次购药日期`、`创建日期`、`任务类型`。

**项目药房 TOP 清单**（可选，xlsx）：
`对应的药店名称`/`药房名称` + 各品种 `TOP`/`项目`/`重点` 列 + `城市`。用于分品种判定药房是项目/非项目，并给药房打城市标签。

> 支持品牌：泰瑞沙 / 优赫得 / 利普卓 / 英飞凡 / 沃瑞沙 / 凡舒卓 / 荃科得。用药间隔（每盒天数）内置，可在 `dtp_engine.DAYS_PER_BOX` 调整。

## 重要口径边界

- **DOT 口径**：严格回滚1年窗口（如 2025-07-01 ~ 2026-06-30），指标 = 窗口内购买总数量 ÷ 去重患者数（盒/人），不换算月数；老患 = 窗口起始前首购，新患窗口 = 首购落在窗口内。
- **留存率两套口径**：口径1（仅购药时间）会因一次购多盒、次月不买而非单调递减（真实特征）；口径2（盒数覆盖）把囤药者的在治期计入，更平滑单调、更贴近真实在治率。
- **脱落率 B 倍数**：填 3 时≈「药吃完后再空 2 个周期还没来买」；算法实际从末次购药日起算， threshold = 倍数 × 每盒天数。
- **品牌自丰富映射**：`brand_aliases.json` 随每次上传自动学习新「通用名→品牌」并落盘；识别不出的写入 `unresolved` 字段。
- **两表无共同患者主键**：销售表会员号与随访表患者编码格式不同，无法直接对患者关联。故品牌层面关联为主；患者级以「姓名」为主键、药房名作软校验（归一化后对齐→高置信，否则→近似），同名歧义时置信度有限。
- **药房名实体对齐**：销售表(如"成都晟德药房")与随访表(如"四川省晟德药房有限公司")命名体系不同。引擎内置归一化(去法人壳词/大药房→药房)做软对齐；可人工维护 `pharm_map.json`(销售简称→随访全称)，自动发现的候选写在 `pharm_map.auto.json` 供抄录。
- **脱落原因分类框架（修正要点）**：改用 `churn_logic` 固定框架——① 先定分类(10细类+一级：医生相关/患者相关/其他)，不反推口径；② `classify_reason` 四原则顺序即优先级：精确枚举→关键词兜底→医生相关→患者相关→其他(统一兜底)；③ "停药"必须配合"医嘱/遵"判定，避免医生原因漏给"患者自主停药"；④ 空原因单独计数，不混入"其他原因"。**归因维度仅用「细分类 + 一级分类」，不使用可控/不可控标签**（避免把医嘱停药等简化为运营可干预，也避免单一标签掩盖真实归因）。
- **随访覆盖缺口**：若随访表某品种为 0 行，跨表该品种原因为空，属覆盖缺口非计算错误。
- **项目药房判定**：优先读取上传的 TOP 清单；无则使用销售底表「角色」列；均无则默认全部药房为项目，城市留空。

## 文件结构

```
app.py             # Streamlit 自助分析 Web App（入口）
dtp_engine.py      # 分析引擎（留存/脱落/DOT/新患/医院医生药房/跨表关联/自丰富品牌映射/Word报告）
churn_logic.py     # 脱落原因清洗核心（10细类+一级+classify_reason四原则+双视角明细），纯逻辑可移植
run_dtp.py         # 本地批量运行：读固定路径数据 → 输出 Excel（便于无界面跑全量）
brand_aliases.json # 品牌别名映射表（通用名→品牌），随运行自动丰富
pharm_map.json     # 药房名实体对齐别名表（人工维护：销售简称→随访全称），可选
pharm_map.auto.json # 自动发现的药房对齐候选（供人工确认后抄入 pharm_map.json）
requirements.txt   # 依赖
README.md          # 本文档
```
